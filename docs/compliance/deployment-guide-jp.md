## 1. 導入前の確認事項

### 1.1 本製品の位置づけ

TrustGate for Workforce は **AIデータ保護（DLP）製品** です。従業員が業務で利用する外部AIサービス（ChatGPT、Gemini、Claude.ai等）への入力テキストを検査し、個人情報・機密情報の漏洩を防止します。

**本製品は「従業員監視ツール」ではありません。**
- 検査対象は指定されたAIサービスのドメインのみ（全ウェブ活動ではない）
- 入力テキストの原文は保存しません（SHA256ハッシュのみ）
- 目的はデータ漏洩防止であり、勤務態度の評価ではありません

### 1.2 前提条件（責務の分解）

TrustGate for Workforce はブラウザ経由のAI利用のみを検査対象とします。ネイティブアプリ（ChatGPTデスクトップアプリ、Claudeアプリ等）経由のAI利用は対象外です。

| レイヤー | 責務 | 担当製品 |
|---------|------|---------|
| ネットワークレベル | AI SaaSへのアクセス制御（ブロック/許可） | ファイアウォール / プロキシ（Zscaler等） |
| デバイスレベル | AIネイティブアプリのインストール制御（禁止/許可） | MDM（Intune, Jamf等） |
| ブラウザレベル | AI SaaS利用時の入出力検査・送信阻止 | **TrustGate for Workforce** |

**導入前に以下を実施してください：**

1. MDM（Intune/Jamf等）でAIネイティブアプリ（ChatGPT, Claude, Gemini等）のインストールを制限する
2. 必要に応じて、ファイアウォールでapi.openai.com等へのアクセスをブロックする（アプリがインストールされていても通信を遮断）
3. TrustGate for Workforce はブラウザ（Chrome）経由のAI利用のみを監視する

### 1.3 関連法規

| 法規 | 関連条項 | 要求事項 |
|------|---------|---------|
| 個人情報保護法（APPI） | 第17条（利用目的の特定）、第18条（取得時の通知） | 利用目的の具体的な特定と事前通知 |
| 労働基準法 | 第89条（就業規則の記載事項）、第90条（意見聴取） | 就業規則の改定、労働者代表への意見聴取 |
| 労働基準法 | 第106条（周知義務） | 改定後の就業規則の全従業員への周知 |
| 個人情報保護法 | 第28条（外国第三者提供） | 国外へのデータ移転制限（該当する場合） |

### 1.3 導入判断基準

以下の全てに該当する場合、導入を推奨します：

- [ ] 従業員がChatGPT等の外部AIサービスを業務で利用している
- [ ] 顧客情報・機密情報がAIサービスに入力されるリスクがある
- [ ] 情報セキュリティポリシーにAI利用に関する規定がない、または不十分である
- [ ] 経営層（CISO/CAIO/CIO）の導入承認が得られている

## 2. 法務・労務手続き

### 2.1 全体スケジュール（目安）

```
Week 1:  導入決定、法務・人事・IT部門のキックオフ
Week 2:  就業規則改定案の作成、従業員通知書の作成
Week 3:  労働者代表への意見聴取（過半数組合 or 従業員代表）
Week 4:  就業規則の届出（労働基準監督署）、従業員への周知
Week 5:  技術導入・テスト（Shadow モード）
Week 6:  Shadow モードの結果評価、ポリシー調整
Week 7:  Enforce モードへ移行、運用開始
```

### 2.2 就業規則の改定

**根拠：** 労働基準法第89条（常時10人以上の労働者を使用する事業場）

就業規則に以下の条項を追加します。テンプレートは `templates/work-rules-amendment-jp.md` を参照してください。

**追加する章：** 「AI利用およびデータ保護に関する規定」

主な記載事項：
1. AIデータ保護システムの導入目的
2. 検査の対象（外部AIサービスへの入力テキスト）
3. 検査の方法（ブラウザ拡張による自動検査、原文は保存しない）
4. 検出時の対応（マスク、警告、ブロック）
5. 記録の保存期間と管理方法
6. 従業員の権利（自身の検出記録へのアクセス権）
7. 問い合わせ窓口

### 2.3 労働者代表への意見聴取

**根拠：** 労働基準法第90条

| 手順 | 内容 | 期限 |
|------|------|------|
| ①代表者の確認 | 過半数組合がある場合はその組合。ない場合は過半数代表者 | Week 2 |
| ②改定案の提示 | 就業規則改定案 + 導入の趣旨説明資料を提示 | Week 2 |
| ③意見聴取 | 代表者から書面で意見を受領（同意は不要、意見を聴くことが要件） | Week 3 |
| ④意見書の添付 | 労基署届出時に意見書を添付 | Week 4 |

> **注意：** 意見聴取は「同意」を求めるものではありません。反対意見があっても、意見を聴取した事実があれば手続き上は有効です。ただし、円滑な導入のためには丁寧な説明と理解の獲得を推奨します。

### 2.4 従業員への通知

**根拠：** 個人情報保護法第18条、労働基準法第106条

通知は以下の2段階で行います：

**① 全体説明会の開催**（推奨）
- 導入の目的と背景
- 検査の仕組み（何が検査され、何が保存されるか）
- 従業員の権利と問い合わせ先
- Q&A

**② 書面での通知（必須）**
- テンプレート `templates/employee-notice-jp.md` を使用
- 全従業員に配布し、受領確認を取得
- 社内イントラネット・掲示板にも掲載

### 2.5 労働基準監督署への届出

就業規則改定後、遅滞なく所轄の労働基準監督署に届出：
- 改定後の就業規則
- 労働者代表の意見書

### 2.6 個人情報保護法対応

| 対応事項 | 内容 |
|---------|------|
| 利用目的の特定 | 「従業員が業務で使用する外部AIサービスへの入力データを検査し、個人情報および機密情報の漏洩を防止するため」 |
| プライバシーポリシー更新 | 社内向けプライバシーポリシーに上記目的を追記 |
| 安全管理措置 | ハッシュのみ保存、ローカル処理、アクセス権限管理等を文書化 |
| 保有個人データの公表 | 検出ログ（ユーザーID、検出カテゴリ、タイムスタンプ）を保有個人データとして公表 |
| 開示請求対応 | 従業員から自身の検出記録の開示請求があった場合の対応手順を整備 |

### 2.7 PrivacyMark保有企業の追加対応

Pマーク（JIS Q 15001）認証企業は以下も必要：
- PMS（個人情報保護マネジメントシステム）文書の更新
- リスク分析表への追記
- 内部監査項目への追加

## 3. 技術導入

### 3.1 構成概要

```
従業員PC
├── ブラウザ（Chrome）
│   └── TrustGate Extension（MV3）
│       → 対象ドメイン: chatgpt.com, gemini.google.com, claude.ai 等
│       → 入力テキストをlocalhost HTTPで送信
└── TrustGate Desktop Agent
    └── localhost:8787/v1/inspect
        → テキスト検査（Regex + ポリシー評価）
        → 検出結果のみ記録（原文は保存しない）
        → [Pro] TrustGate Cloud（日本リージョン）へ統計・ハッシュのみ送信
```

### 3.2 導入手順

#### Step 1: Desktop Agent のインストール

```bash
# Windows（管理者権限で実行）
aigw.exe service install
aigw.exe service start

# macOS
brew install trustgate/tap/aigw   # (将来対応)
# または直接バイナリ配置
sudo cp aigw /usr/local/bin/
```

#### Step 2: 設定ファイルの配置

`agent.yaml` を配置します。以下は推奨設定：

```yaml
version: "1"
mode: standalone  # Pro版の場合: managed

listen:
  host: 127.0.0.1
  port: 8787

identity:
  mode: header
  headers:
    user_id: X-TrustGate-User
    role: X-TrustGate-Role
    department: X-TrustGate-Department
  on_missing: anonymous
  anonymous_role: guest

detectors:
  pii:
    enabled: true
    patterns:
      email: true
      phone: true
      mobile: true
      my_number: true
      credit_card: true
      address: true
      date_of_birth: true
      bank_account: true
  injection:
    enabled: true
    language: [ja, en]
  confidential:
    enabled: true
    keywords:
      critical: ["極秘", "社外秘", "CONFIDENTIAL", "TOP SECRET"]
      high: ["機密", "内部限定", "INTERNAL ONLY", "取扱注意"]

policy:
  source: local
  file: ./policies.yaml

audit:
  mode: local
  path: ./audit.db
  retention_days: 90  # 個人情報保護法の最小限原則に基づく

logging:
  level: info
  format: json
```

#### Step 3: ポリシーの初期設定

**重要：初期導入は全ポリシーを Shadow モードで開始します。**

```yaml
version: "1"

policies:
  # 全て mode: shadow で開始
  - name: block_injection_critical
    phase: input
    when:
      detector: injection
      min_severity: critical
    action: block
    mode: shadow    # ★ 初期は shadow

  - name: mask_pii_input
    phase: input
    when:
      detector: pii
      min_severity: high
    action: mask
    mode: shadow    # ★ 初期は shadow

  - name: block_confidential_input
    phase: input
    when:
      detector: confidential
      min_severity: critical
    action: block
    mode: shadow    # ★ 初期は shadow
```

#### Step 4: ブラウザ拡張のデプロイ

**企業管理端末（Google Workspace / Intune）の場合：**
- Chrome管理コンソールからforced installで配布
- 従業員個人での許可/拒否は不要（ただし事前通知は必須）

**手動インストールの場合：**
- Chrome Web Storeから「TrustGate AI Guard」をインストール

#### Step 5: 動作確認

```bash
# Agent の動作確認
curl -s http://localhost:8787/v1/health | jq .

# テスト検査
curl -s http://localhost:8787/v1/inspect \
  -H "Content-Type: application/json" \
  -d '{"text":"テスト入力です"}' | jq .
```

## 4. 運用開始

### 4.1 Shadow モード期間（推奨：2〜4週間）

Shadow モードでは、検出・ポリシー評価は行われますが、実際のブロック・マスクは行いません。「もし enforce だったら何が起きたか」がログに記録されます。

**この期間で確認すること：**

| 確認項目 | 合格基準 | 対応 |
|---------|---------|------|
| 誤検知率 | 10%以下 | ポリシー/検出器の調整 |
| 業務影響範囲 | ブロック対象が1日10件以下 | 閾値・ホワイトリストの調整 |
| 検出漏れ | テストケースの90%以上を検出 | パターンの追加 |
| パフォーマンス | 検査レイテンシ10ms以下 | 設定の最適化 |

### 4.2 Enforce モードへの移行

Shadow モードの評価結果を確認し、段階的に enforce に切り替えます：

```
Step 1: injection検出のみ enforce（最もリスクが高く、誤検知が少ない）
Step 2: PII検出を enforce（mask動作なので業務停止しない）
Step 3: confidential検出を enforce（最後。誤検知の影響が大きいため）
```

各 Step の間に最低1週間の観察期間を設けます。

### 4.3 従業員への運用開始通知

enforce モード移行時に再度通知します：

```
件名：AIデータ保護システムの運用開始について

従業員各位

[日付] より、AIデータ保護システム（TrustGate）の運用を開始いたします。

これにより、業務で使用する外部AIサービス（ChatGPT、Gemini、Claude等）への
入力時に、以下の保護が自動的に適用されます：

・個人情報（電話番号、メールアドレス等）→ 自動マスク
・機密情報（社外秘等のキーワード）→ 送信ブロック
・不正な指示（プロンプトインジェクション）→ 送信ブロック

なお、入力テキストの原文は一切保存されません。

ご不明点は情報セキュリティ部門（内線: XXXX）までお問い合わせください。
```

## 5. 継続運用

### 5.1 定期レビュー

| 頻度 | 内容 | 責任者 |
|------|------|--------|
| 毎週 | 検出件数・ブロック件数の確認 | IT部門 |
| 毎月 | 誤検知レポートの確認、ポリシー調整 | 情報セキュリティ部門 |
| 四半期 | 経営層向けレポート（AI利用状況、リスク傾向） | CISO/CAIO |
| 年次 | ポリシー全体の見直し、法改正への対応 | 法務 + IT |

### 5.2 従業員からの問い合わせ対応

| 問い合わせ種別 | 対応 |
|--------------|------|
| 「正当な入力がブロックされた」 | ホワイトリスト追加を検討。`POST /v1/audit/{id}/feedback` でフィードバック記録 |
| 「自分の検出記録を見たい」 | 個人情報開示請求として対応。監査ログからユーザーIDで抽出 |
| 「監視されていると感じる」 | 目的（データ保護）と仕組み（原文非保存）を再説明 |

### 5.3 インシデント対応

機密情報の大量漏洩が検出された場合：

1. 即時：対象ユーザーの利用を一時停止（ポリシーで対応）
2. 1時間以内：情報セキュリティ責任者に報告
3. 24時間以内：影響範囲の調査、経営層への報告
4. 必要に応じて：個人情報保護委員会への報告（漏洩が1,000人超の場合、72時間以内）
